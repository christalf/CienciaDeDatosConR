1 Caching in literate statistical programming.

In the literate statistical program framework, we have the following situation:

  • Our paper has text and code.

  • The code chunks do something with the data and we store the results and the code in some central, in some database.

  • We have then the paper the figures and tables generated from the materials stored in the database.

There are some R packages that implement this process.

  • One is the cacheSweave package which we won’t talk about, but it interacts with Sweave.

  • Another one is the cacher package, which is a stand alone package.

1.1 The cacher package.

This package assumes that we have some code in an R file. And, basically, it reads the code, evaluates it and stores the results in a key-value database.

The R expressions are all given SHA-1 hash values so that changes can be tracked and code can be reevaluated if necessary.

So, the idea is that we put everything, the code and data, in a package that we could just give to someone.

  • We call this a cacher package.
  • Other people can clone the analysis and look at subsets of code or inspect specific data objects.
  • One assumption that we make here is that other people do not necessarily have the resources that we have.
    • So they may not want to run the entire Markov chain Monte Carlo simulation that we did to get the posterior distribution, or the histogram we’ve got at the end.
    • They might just want to look at specific end products of that.
    • And then maybe, when they see the end products, they decide to look at the whole MCMC.
  • But the idea is that one can kind of peel the onion a little bit rather than just go straight to the core.

1.1.1 The conceptual model.

  • We have some code that goes in. There’s data that goes in. And out comes results.

1.1.2 Using cacher as an author.

As author, the idea of cacher is that it parses the source file as follows:

  • It creates directories and sub-directories for storing things.
  • Then it evaluates each expression and stores any resulting R objects in the cache database. In other words, as it runs expressions, it caches the results in the database.
    • If we run it multiple times and it happens that an expression hasn’t changed, it won’t run it again, it just loads the results back from the database.
    • If an expression does not creat any R objects (i.e., there’s nothing to cache), it adds the expression to the list of expressions where evaluation needs to be forced.
    • It also writes out a bunch of metadata for each expression, in a metadata file.

Once it has evaluated our whole analysis (this might take a long time), all the intermediate results are stored in the database and we can wrap that up into a package.

  • That’s the cachepackage function does.
    • It creates a cacher pakage, storing:
      • source file
      • cached data objects
      • metadata
    • The package just becomes a zip file, and it can be distributed to others.
    • Readers can unzip the file and immediately investigate its contents via cacher package.

1.1.3 Using cacher as a reader: clonecache() function.

The package will have an identifier like the SHA-1 in the following example:

  • Everything about the research or article wrote by the author (data and code) can be found in the package with that identifier string.
  • It’s ugly, but we can just use the first four characters, that’s usually enough to identify the package itself.
  • So if we’re reading something like this in a journal, we could take that string and clone the whole thing using the clonecache() function.
    • This way, we can see what files were cached there and go through the analysis.
    • Some local directories are created.
    • The source code files and the metadata are downloaded.
      • The data themselves, instead, are not downloaded initially. It could be a lot!
      • But references to the various data objects that have been created are loaded, so that they can be lazy-loaded into the environment on demand. Once we want to look at something, then it gets downloaded onto our computer.

1.1.3.2 Tracing code backwards: objectcode() function.

We could for instance be interested in inspecting the data object, see what went into the creation of that object.

We use for this the objectcode() function.

  • This funciont shows us the specific lines of code that were used to generate that object.
  • There may be different types of objects in the analysis. We can hone in on some of them and look at the specific coded that generated that object.

1.1.3.3 Running code: runcode() function.

We can also execute the code with the runcode() function.

  • It runs the code, but in a way that it just loads things from the database without actually executing code by default.
    • Expressions that results in an object being created are not run and the resulting objects are lazy-loaded into de workspace.
    • Expressions not resulting in objects are evaluated.
  • So if we just want to run and see some results at the end as a first cut, then it’s not going to do everything from scratch by default. Thus, it’s a lot faster on the first go.
  • Then, of course, things that aren’t stored in the database will have to be generated (for example, a plot isn’t stored in the database and it’s still going to have be generated).

1.1.3.4 Checking code and objects: checkcode() and checkobjects() functions.

We can of course just force to run everything from scratch and do all the calculations from scratch.

This is what the checkcode() function does.

  • It evaluates all the expressions from scratch. It doesn’t load anything from the database, there’s no lazy-loading, it runs all straightaway.
  • Then the evaluations are checked against stored results.
    • If a function creates an object, when the author created the package the signature for that object was stored so that if we create the object again, it’ll check the signatures to see if they match.
    • Of course we’ve got to set things like random generator seeds for this kind of thing to work.

There’s another function, checkobjects(), that verifies the integrity of each individual data object, checking for any possible corruption of data (i.e. in transit).

  • It basically checks the signatures.

1.1.3.5 Inspecting data objects: loadcache() function.

We can also load and look at specific data objects, with the loadcache() function.

  • This function just loads pointers to specific data objects that may be stored on a server or something like that.
  • Once we print something out to the screen or we want to make a plot, any time we access the object, it will transfer it from wherever it’s coming from.
  • So, in our example, the cities transferring occurs from the cache database file (we can see the signature there of that object) and then it prints, it shows us what it is.
  • Once it’s loaded into our system, it’s loaded for good. We don’t have to do it every single time, just the first time.

1.1.4 Summary.

  • With the cacher package, authors can create cache packages from their data analysis for distribution.

  • Then readers can use the cacher package to ispect others’ data analyses by examining cached computations (download the analysis and look at specific objects or maybe rerun the whole thing).

  • cacher package is mindful of readers’ resources.

    • It efficiently loads only those data objects that are needed.

    • In other words, there’s flexibility in terms of not having to recreate the entire environment if it’s a very complicated type of thing.

2 Reproducible research case study: air pollution.

2.1 What Causes PM to be Toxic?

  • PM is composed of many different chemical elements

  • Some components of PM may be more harmful than others

  • Some sources of PM may be more dangerous than others

  • Identifying harmful chemical constituents may lead us to strategies for controlling sources of PM

2.2 NMMAPS

  • The National Morbidity, Mortality, and Air Pollution Study (NMMAPS) was a national study of the short-term health effects of ambient air pollution

  • Focused primarily on particulate matter (\(PM_{10}\)) and ozone (\(O_3\))

  • Health outcomes included mortality from all causes and hospitalizations for cardiovascular and respiratory diseases

  • Key publications

    • http://www.ncbi.nlm.nih.gov/pubmed/11098531

    • http://www.ncbi.nlm.nih.gov/pubmed/11354823

  • Funded by the Health Effects Institute

2.3 NMMAPS and Reproducibility

  • Data made available at the Internet-based Health and Air Pollution Surveillance System (http://www.ihapss.jhsph.edu)

  • Research results and software also available at iHAPSS

  • Many studies (over 67 published) have been conducted based on the public data http://www.ncbi.nlm.nih.gov/pubmed/22475833

  • Has served as an important test bed for methodological development

2.4 What Causes Particulate Matter to be Toxic?

http://www.ncbi.nlm.nih.gov/pmc/articles/PMC1665439/

  • Lippmann et al. found strong evidence that Ni modified the short-term effect of \(PM_{10}\) across 60 US communities

  • No other PM chemical constituent seemed to have the same modifying effect

  • To simple to be true?

2.5 A Reanalysis of the Lippmann et al. study

http://www.ncbi.nlm.nih.gov/pmc/articles/PMC2137127/

  • Reexamine the data from NMMAPS and link with PM chemical constituent data

  • Are the findings sensitive to levels of Nickel in New York City?

2.6 Does Nickel Make PM Toxic?

  • Long-term average nickel concentrations appear correlated with PM risk

  • There appear to be some outliers on the right-hand side (it turns out that they are three counties in New York City - NYC is composed of five counties)

  • The positive slope indicates a positive correlation, so that more nickel is associated with a greater risk of mortality from PM.

  • And the regression line appears as statistically significant (\(p < 0.01\))

  • This was the conclusion of the original paper.

  • However, if we just remove the outlying points (in particular the points that are in New York City where nickel levels are very high) and we redo the regression line (the blue one), we see that the adjusted regression line is no longer statistically significant (\(p < 0.31\)).

  • So those three points to the right we took off, are called high leverage points. And the regression line can be very sensitive to high leverage points.

  • There were 60 communities in the original study.

  • Here, we went through each one and just redid the analysis by removing one community. Just to see if there were any particular communities that the analysis was sensitive to.

  • And we can see that all the black dots represent the slopes of the regression line when we remove a given community.

  • Because they’re all clustered around the same value, we can see that for the most part the regression line is not sensitive to removing any individual community from the dataset.

  • However, the red dot and red line at the very bottom indicates the slope estimate and the 95% confidence interval when we remove New York.

    • The slope estimate goes down quite a bit and the confidence intervals goes up (it gets quite a bit wider than all of the other dots).

    • And so, that’s another indication that the slope estimate of that regression line is very sensitive high leverage points.

2.7 What Have We Learned?

  • New York does have very high levels of nickel and vanadium, much higher than any other US community

  • There is evidence of a positive relationship between Ni concentrations and \(PM_{10}\) risk

  • The strength of this relationship is highly sensitive to the observations from New York City

  • Most of the information in the data is derived from just 3 observations

  • Reproducibility of NMMAPS allowed for a secondary analysis (and linking with PM chemical constituent data) investigating a novel hypothesis (Lippmann et al.)

  • Reproducibility also allowed for a critique of that new analysis and some additional new analysis (Dominici et al.)

  • Original hypothesis not necessarily invalidated, but evidence not as strong as originally suggested (more work should be done)

  • Reproducibility allows for the scientific discussion to occur in a timely and informed manner

  • This is how science works

3 Reproducible research case study: high-throughput biology.

This section is a Keith Baggerly lecture, who is a professor in the Biostatistics and Bioinformatics Department at the MD Anderson Cancer Center, about a case that happened with some researchers at Duke University regarding using high dimensional genomic signatures in cancer outcomes.

We can find the lecture at this link:

http://videolectures.net/cancerbioinformatics2010_baggerly_irrh/

  • The data and analysis being dissected relate to the now famous Duke/Potti controversy described by:
    • the New York Times
    • also 60 minutes
    • and others.
  • The work of Biostatisticians Keith Baggerly and Kevin Coombes ultimately led to the retraction of ten papers by Potti, and the cancellation of clinical trials.
  • A recent Nature article provides a follow-up and guidance for uncovering research misconduct.
  • This is an interesting case study on:
    • the importance of being able to reproduce complicated computational analyses;
    • how some very problematic work from an otherwise productive and potentially useful area of research (personalized chemotherapy prediction) went horribly wrong;
    • and how we might use this experience to promote transparent, open source, and reproducible research in bioinformatics. For example, by using things like Sweave. Another approach is to use, Rmarkdown and Knitr.
